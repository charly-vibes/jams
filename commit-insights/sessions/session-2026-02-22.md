# Session 2026-02-22 — GitHub Authentication

## Goal

Improve the GitHub API usage by adding authentication, raising the rate limit from 60 to 5,000 req/hr and enabling access to private repo commit data.

## What Was Built

### Two login options

**GitHub OAuth (Device Flow)**
The ideal UX: user clicks "Continue with GitHub", a tab opens to `github.com/login/device`, they enter an 8-char code (click to copy), and the app automatically receives a token. No PAT creation needed.

**Personal Access Token**
Fallback for users who prefer it. Paste a `ghp_…` token directly. A link pre-scoped to `repo,read:user` makes token creation fast.

Both paths share `onTokenReceived()` which fetches `/user`, stores auth state in `localStorage`, and auto-fills the username input.

### Cloudflare Worker (`worker.js`)

GitHub's OAuth endpoints (`github.com/login/device/code`, `github.com/login/oauth/access_token`) don't support CORS, so browser `fetch` is blocked. A thin Cloudflare Worker proxies the two endpoints and adds `Access-Control-Allow-Origin: *`.

Deployed at: `https://commit-insights-oauth.sasha-hipervinculos.workers.dev`

The Worker URL is hardcoded as the default in the app and hidden from the UI.

## Issues Encountered

**CORS on GitHub OAuth endpoints**
The first attempt called GitHub's Device Flow endpoints directly from the browser. This failed immediately with a `NetworkError` — GitHub doesn't send CORS headers on `github.com/login/…`. Fixed by routing through the Cloudflare Worker.

**Cloudflare dashboard upload error**
The Worker was initially written with ES module syntax (`export default { … }`). The Cloudflare dashboard uploader rejected it with "does not support projects that require a build process". Fixed by rewriting with Service Worker syntax (`addEventListener('fetch', …)`). Eventually deployed via `wrangler` CLI using a Cloudflare API token to bypass the dashboard OAuth loop.

**Login hanging on "Waiting for authorization"**
After the user approved on GitHub, the app stayed stuck. The polling used `setInterval` and silently swallowed all errors with `catch (_) {}`. Adding `console.log` revealed GitHub was returning `slow_down` (too many requests in the same timeframe, back off to 15s). Since `slow_down` was grouped with `authorization_pending` as "keep polling", the interval never adjusted and GitHub kept rejecting every poll. Fixed by switching to recursive `setTimeout` so the delay can be updated dynamically from the response.

## Bug Fixed

`renderDashboard(userData)` had a parameter that shadowed the module-level `userData` variable. Since all call sites pass no arguments, the parameter was always `undefined`, which would throw on `userData.avatar_url`. Fixed by removing the unused parameter.

## Key Decisions

- Worker URL hardcoded as default, not exposed in UI — no configuration needed for the common case
- `localStorage` persists both the token and the user object across sessions
- Device code clicks copy to clipboard and briefly show "Copied!" as feedback
- All `api.github.com` fetch calls use `getHeaders()` which adds the `Authorization` header when a token is present
