# Session 2026-03-11 — Initial Setup from External Prototype

## Overview

Ported a pre-built Marginalia PWA (generated externally as a single-file preview + zipped multi-file PWA) into the jams repo structure. The app addresses a real workflow pain point: capturing post-it notes from physical books and turning them into structured Obsidian-compatible Markdown notes.

## Origin

The app was generated from a prompt describing a note-taking workflow problem:

- While reading, place post-it notes on book pages to mark interesting passages
- After finishing the book, go back through and create proper notes with full context
- Pain point: books pile up with post-its but notes never get written up
- Solution: photograph pages with post-its, OCR the text, categorize, and export

The source arrived as `files.zip` containing a standalone preview HTML and a `booknotes-pwa.zip` with the separated multi-file PWA.

## Adaptations for Repo Structure

### File Renames
- `app.js` → `script.js` (repo convention — all apps use `script.js`)

### Path Fixes for GitHub Pages
The original PWA used root-absolute paths (`/sw.js`, `/index.html`) which would break when deployed as a subdirectory on GitHub Pages (`charly-vibes.github.io/jams/marginalia/`). Changed to relative paths:
- Service worker registration: `/sw.js` → `./sw.js`
- Service worker cache list: `/index.html` → `./index.html`, etc.
- Manifest `start_url`: `/index.html` → `./`

This is a common gotcha with PWAs on GitHub Pages — the service worker scope is relative to its location, and absolute paths would try to cache from the site root.

## Architecture Notes

### Data Model (IndexedDB)
Three object stores:
- **books** — `{ id, title, author, createdAt, updatedAt }`, indexed by title
- **notes** — `{ id, bookId, text, highlight, pageNum, tags, photoId, createdAt }`, indexed by bookId and highlight type
- **photos** — `{ id, data }` where data is a base64 data URL

Photos are stored separately from notes to avoid loading large blobs when listing/filtering notes.

### OCR Pipeline
- Uses Tesseract.js v5 from CDN
- Worker is lazy-loaded on first use (or pre-loaded from Settings)
- Two capture modes:
  - **Single**: photo → OCR → immediate review modal
  - **Batch**: multiple photos → OCR all → step-through review wizard
- OCR progress displayed per-thumbnail with status badges

### View Routing
SPA-style view switching via CSS class toggling (`.view.active`), no hash routing. The FAB button context-switches between "Add Book" (on books list) and "Capture" (on book detail).

### Export
Markdown export groups notes by highlight type (Important → Quotes → Ideas → Questions → Notes) with emoji headers. Designed to paste directly into Obsidian.

## Design Departures

This app intentionally departs from the repo's grayscale default:
- Parchment/amber/rust color scheme evokes books and paper
- Post-it note colors (yellow, pink, blue, green) for highlight categories
- Serif typography (Playfair Display + Source Serif 4) for a literary feel
- Custom fonts loaded from Google Fonts CDN

This is justified by the spec — the CLAUDE.md says "grayscale by default *unless a specific color scheme is required*" and a book-notes app benefits from the warmth.

## Potential Improvements

- Manual note entry without photo (for when you just want to type a note)
- Bulk operations (delete multiple notes, merge books)
- Photo viewer (tap a note to see the original captured page)
- Multi-language OCR (Tesseract supports many languages, currently hardcoded to English)
- Share API integration for sending exports directly to other apps
