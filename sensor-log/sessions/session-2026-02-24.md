# Session 2026-02-24 — Initial Implementation

## Overview

Full implementation of the SensorLog PWA from scratch, based on `SensorLog-PWA-Spec-v1.0.docx`. Implemented all 8 phases from the spec in a single session.

## Implementation Decisions

### Architecture

**Dual-buffer strategy**: Instead of using a single ring buffer that drains to storage (which would clear display data), we use two structures per sensor:
- `RingBuffer` — circular display buffer (512 samples), never cleared, canvas reads from here
- `pending[]` — grows between drains, transferred to the storage worker every 3 seconds and then cleared

This avoids the tradeoff in the spec's suggested design where `drain()` would reset the ring buffer and lose display continuity.

**Transferable ArrayBuffers**: The drain operation creates a `Float64Array` from `pending[]` and transfers its `.buffer` via `postMessage`. This is zero-copy — the worker takes ownership and the main thread can no longer access it. This minimizes GC pressure on the main thread.

### Storage Worker

The OPFS binary format uses a simple length-prefixed block structure:
```
[4 bytes: header_len][header_JSON][blocks...]
  where each block: [4 bytes: total_block_len][1 byte: sensorId][float64 data...]
```

The `total_block_len` includes the sensorId byte, so `float64_data.byteLength = block_len - 1`.

`createSyncAccessHandle()` is used with explicit `{ at: writePos }` offsets rather than implicit cursor, so writes are always positioned correctly even if the worker receives messages out of order (which shouldn't happen but is safer).

### Sensor Manager

The `SensorManager` implements a dual-path strategy:
1. **Generic Sensor API** (Chrome/Android): pass `{ frequency: hz }` to constructor
2. **DeviceMotion/Orientation fallback** (Safari/Firefox): add event listeners with decimation (1 of every `Math.round(60/hz)` events kept)

For **compass**, there are two fallback paths:
- `deviceorientationabsolute` event (Chrome desktop/Android)
- `webkitCompassHeading` from `deviceorientation` event (iOS Safari)
Both are registered simultaneously; the first to deliver data wins.

For **iOS permissions**: `DeviceMotionEvent.requestPermission()` must be called from a user gesture. This is triggered when the user presses RECORD (not when enabling the toggle), which is a natural user gesture flow.

### Canvas Renderer

Each `SensorChart` manages its own canvas with:
- DPR scaling capped at 2× (prevents excessive memory on high-DPR devices)
- `{ alpha: false }` context (slightly faster compositing)
- `IntersectionObserver` to pause rendering when chart is scrolled off-screen
- Exponential smoothing for Y-axis auto-range with α=0.06 (adjusts gently to new extremes)
- Zero-line rendered as dashed grid even if outside current range

The render loop uses `requestAnimationFrame` with `frameCount % 2 !== 0` skip to target ~30fps from the 60fps rAF cadence.

### Background Keep-Alive

Two components:
1. **Screen Wake Lock** (`navigator.wakeLock.request('screen')`) — both Android and iOS (iOS 16.4+)
2. **Silent audio oscillator** (Android only, skipped on iOS) — 1 Hz oscillator at gain 0.001 keeps the AudioContext alive, preventing Chrome from throttling background tabs

Emergency flush is registered on three events: `visibilitychange`, `pagehide`, and `freeze` (Chrome freeze lifecycle). This covers all the ways a tab can become inactive.

### CSV Export

The export reads the OPFS binary file via the storage worker (worker sends back structured `sensorData` object), then the main thread generates one CSV per sensor. Each CSV has:
- UTF-8 BOM (`\uFEFF`) prepended for Excel compatibility
- Comment lines starting with `#` for metadata
- Columns: `timestamp_ms, field1, field2, ...`

For large datasets, a streaming approach via `showSaveFilePicker` would be preferable, but the current implementation is simpler and handles sessions up to ~100MB fine.

## Issues Encountered

### OPFS availability check

`typeof FileSystemFileHandle !== 'undefined'` is a reasonable check but doesn't confirm `createSyncAccessHandle()` is available. The worker catches errors and posts them back to the main thread, so the UI shows an error toast if OPFS fails at runtime.

### DeviceMotion event rate

The spec says `devicemotion` fires at ~60Hz on most devices. In practice this varies: iOS fires at 60Hz, some Android devices at 50Hz or lower. The decimation calculation `Math.round(60/targetHz)` may be slightly off on non-60Hz devices, resulting in slightly different actual sample rates than requested. This is documented behavior.

### sensor-log vs script.js

The app registers as `sensor-log` in the root `APPS` array. The spec description is pulled from `spec/functionality.md` → `## Purpose` section.

```

├── index.html          (~200 lines) App shell + inline CSS
├── app.js              (~600 lines) Core logic
│   ├── SENSOR_DEFS     Sensor configurations

│   ├── SensorManager   Sensor start/stop/fallback
│   ├── SensorChart     Canvas renderer per sensor
│   ├── BackgroundKeepAlive  Wake lock + audio
│   └── App             Main coordinator

├── sw.js               (~25 lines)  Service Worker

    ├── icon.svg        Waveform icon
    └── icon-192.svg    Maskable variant
```
